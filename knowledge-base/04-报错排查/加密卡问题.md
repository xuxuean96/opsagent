# 加密卡问题

## 组成

- Windows 服务端：`KmPermitServer.exe`
- Linux 服务端：`KmPermitServer_Linux`
- Java 客户端：`KmEncryptCardClient.jar`
- 原生客户端：`GetProductInfo.exe` / `libKMPMD.so`

## 核心参数

- 端口通常是 `5600`
- `soft.id` 用来区分产品
- `productInfoNo` 常用 `7`
- `mode` 可分 `session` 和 `long`

## 常见问题

- 服务起不来
- 客户端连不上
- 授权不足
- 授权过期
- 硬件或驱动不匹配
- 加密卡服务端启动成功，但 eCol 或 CLOUD 仍提示不可用
- 加密卡进程存在，但客户机仍然连不上
- 加密卡启动失败，实际是已有进程存在或端口被占用

## 机器人优先追问

1. 服务端是 Windows 还是 Linux
2. `mode` 用的是哪种
3. `soft.id` 是多少
4. 日志里第一条错误是什么
5. 端口 `5600` 是否已监听

## 服务端正常但 eCol/CLOUD 仍不可用

如果加密卡服务端已经启动成功，端口也能连通，但 eCol 或 CLOUD 仍然无法使用，优先怀疑应用程序里配置的加密卡服务地址不对。

### 判断条件

- `KmPermitServer` 进程存在
- 服务端日志没有启动失败
- 服务端端口已经监听
- 使用独立客户端测试服务端可以连通
- 只有 eCol 或 CLOUD 内部功能仍报授权、加密卡连接或许可不可用

### 优先检查

1. eCol/CLOUD 配置里的加密卡服务器 IP 是否指向当前服务端
2. 端口是否和服务端一致，默认通常是 `5600`
3. CLOUD 的客户端模式是否正确，例如 `kmwp.encryption.EncryptionCard.mode`
4. 如果服务端 IP 变更过，确认应用配置没有仍指向旧地址
5. 如果是多网卡或服务器迁移，确认配置中使用的是客户端可访问的 IP

### 处理建议

- 先用独立加密卡客户端直连服务端验证
- 再检查 eCol/CLOUD 的加密卡地址配置
- 修改配置后重启对应应用服务
- 复测授权功能，并对比应用日志和 `KmPermitServer.log`

## 进程存在但客户端连不上

如果服务端机器上已经能看到 `KmPermitServer` 进程，日志也显示启动成功，但客户机、eCol 或 CLOUD 仍然连不上，应优先检查网络连通和防火墙。

### 优先检查

1. 客户机能否 `ping` 通加密卡服务端 IP
2. 客户机能否访问服务端端口，默认通常是 `5600`
3. 服务端防火墙是否放行加密卡端口
4. 客户机本机防火墙或安全软件是否拦截出站连接
5. 服务端是否监听在正确网卡和正确 IP 上
6. 如果跨网段，确认路由、ACL、安全组或网闸策略是否放行

### 判断方式

- 服务端本机能连，客户机不能连：优先查服务端防火墙、客户机防火墙、网络策略
- 部分客户机能连，部分不能连：优先查客户机侧防火墙、网段策略或本机安全软件
- 所有客户机都不能连，但服务端进程存在：优先查服务端端口监听和防火墙入站规则

## 启动失败先查已有进程

如果加密卡启动失败，先不要直接重装或覆盖文件，应先检查是否已经存在加密卡进程。

### 优先检查

1. 是否已有 `KmPermitServer` 进程
2. 默认端口 `5600` 是否已被占用
3. 是否重复启动了多个服务端实例
4. 旧进程是否僵死但仍占用端口

### 处理建议

- 如果已有正常进程，先不要重复启动
- 如果旧进程异常，先停止旧进程再重新启动
- 如果端口被其他程序占用，修改端口或释放占用
- 重启后查看 `KmPermitServer.log`，确认是否是真正启动成功
