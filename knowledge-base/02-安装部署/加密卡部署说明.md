# 加密卡部署说明

## Windows 服务端

- 运行 `KmPermitServer.exe`
- 端口一般为 `5600`
- 需要授权文件和正确驱动

## Linux 服务端

- 运行 `KmPermitServer_Linux`
- 依赖库要齐
- `LD_LIBRARY_PATH` 要正确
- 授权文件和硬件状态要匹配
- 启动失败时要直接执行服务端程序或启动脚本，看是否缺少运行时

## 部署检查

1. 服务是否启动
2. 端口是否监听
3. 授权文件是否存在
4. 客户端模式是否匹配
5. 日志是否有报错

## 启动失败检查

加密卡启动失败时，先检查是否已经存在加密卡进程，避免重复启动导致误判。

需要检查：

- 是否已有 `KmPermitServer` 进程
- 默认端口 `5600` 是否已被占用
- 是否存在旧进程未退出
- 是否重复启动了多个实例

如果已有进程且端口正常监听，说明服务端可能已经启动成功，此时应转入客户端连通性排查。

## Linux 运行时检查

Linux 加密卡服务端启动失败时，先执行 `KmPermitServer_Linux_Server.sh` 或直接执行 `KmPermitServer_Linux`，观察是否缺少运行库。

常见方向：

- 缺少 `.so` 依赖
- `GLIBCXX` 或 `libstdc++` 版本不满足
- `LD_LIBRARY_PATH` 未包含服务端 `lib` 目录

运行时问题解决后，再继续检查授权文件、端口、防火墙和客户端配置。

## 应用侧地址检查

加密卡服务端启动成功不代表 eCol 或 CLOUD 一定能使用。应用程序还必须配置正确的加密卡服务端地址和端口。

需要检查：

- eCol/CLOUD 配置中的加密卡服务器 IP
- 加密卡服务端口，通常为 `5600`
- CLOUD 客户端模式配置，例如 Java 客户端或原生客户端模式
- 配置修改后是否已重启应用服务

典型现象：

- 服务端日志正常
- 独立客户端能连上
- eCol/CLOUD 仍报授权失败或加密卡不可用

此时优先改应用侧配置，不要先重装加密卡服务端。

## 防火墙检查

如果服务端进程存在、端口也已监听，但客户机连不上，需要检查客户机和服务端两侧防火墙。

重点检查：

- 服务端防火墙是否放行加密卡端口
- 客户机防火墙或安全软件是否拦截访问
- 跨网段时网络策略是否放行
- 服务端监听 IP 是否是客户机可访问的地址
