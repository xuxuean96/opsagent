# 加密卡问答样例

## Q1：加密卡服务启动了，但 CLOUD 还是连不上

优先检查 CLOUD 里配置的加密卡服务端 IP、端口和客户端模式。
如果独立客户端能连上服务端，说明服务端优先判断率正常，不要先重装服务端。

检查顺序：

1. `KmPermitServer` 进程是否存在
2. 端口 `5600` 是否监听
3. 独立客户端能否连通
4. CLOUD 配置里的加密卡 IP 是否正确
5. `kmwp.encryption.EncryptionCard.mode` 是否匹配
6. 修改后是否重启 CLOUD 服务

## Q2：加密卡进程有，但客户机连不上

优先查防火墙和网络策略。

检查顺序：

1. 客户机能否 ping 通服务端
2. 客户机能否访问服务端 `5600` 端口
3. 服务端防火墙是否放行端口
4. 客户机防火墙或安全软件是否拦截
5. 跨网段时路由、ACL、安全组是否放行

## Q3：加密卡启动失败怎么办

先查是否已经有 `KmPermitServer` 进程或端口已被占用。
如果已有正常进程，不要重复启动。如果旧进程异常，先停止旧进程再启动。

## Q4：服务端正常，但 eCol 不能用

先确认 eCol 配置的加密卡服务器地址是否正确，尤其是服务端 IP 变更、双网卡、服务器迁移后。
如果 eCol 指向旧 IP，即使加密卡服务端正常，eCol 也会不可用。

## Q5：怎么判断是授权问题还是网络问题

- 端口连不上：优先网络、防火墙、服务监听
- 端口能连但登录失败：优先授权文件、节点数、soft.id
- 独立客户端成功但 CLOUD 失败：优先 CLOUD 配置
- 服务端启动失败：优先进程、端口占用、依赖库、授权文件
