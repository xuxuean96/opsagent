# 服务启动失败-运行环境检查

## 适用场景

eCol、CLOUD、DDB、KMVue、加密卡等服务或组件无法启动，或者启动后马上退出、报连接失败、报缺少库文件。

## 核心判断

不要只查业务配置。服务无法启动时，必须先确认该软件自己的运行环境是否满足要求。

## 典型现象

- CLOUD 启动失败
- Linux 上 DDBExec 执行失败
- KMVue 的 `kmconvertservice` 启动失败
- Linux 加密卡服务端启动失败
- KMVue Linux 环境报加密卡连接不上，但实际是 `kmconvertservice` 缺少运行时

## 可能原因

| 优先级 | 原因 | 判断依据 |
|---|---|---|
| P1 | Java 环境缺失或版本不对 | CLOUD 相关 jar 启动失败 |
| P1 | Linux DDBExec 缺少运行库 | 直接执行 `DDBExec` 报 so 缺失 |
| P1 | KMVue `kmconvertservice` 缺少运行时 | 转换服务无法启动，可能误报加密卡连接不上 |
| P1 | Linux 加密卡服务端缺少运行库 | 执行 `KmPermitServer_Linux_Server.sh` 或服务端程序时报依赖错误 |
| P2 | 环境变量未配置 | `LD_LIBRARY_PATH`、PATH、Java 路径不正确 |

## 先问用户

1. 哪个服务无法启动：eCol、CLOUD、DDB、KMVue、加密卡
2. 操作系统是 Windows 还是 Linux
3. 是启动失败、启动后退出，还是业务功能不可用
4. 直接执行对应程序时的报错是什么
5. 日志里第一条运行时或依赖错误是什么

## 检查步骤

### CLOUD

1. 检查 Java 是否安装
2. 检查 Java 版本是否符合系统要求
3. 检查启动脚本使用的是哪个 Java
4. 查看 CLOUD 启动日志中是否有 `java`、`class`、`jar`、`UnsupportedClassVersionError` 等错误

### Linux DDBExec

1. 进入 DDBExec 所在目录
2. 直接执行 `./DDBExec`
3. 如果报缺少 `.so`、`GLIBCXX`、`libstdc++`，先补运行库
4. 检查 `LD_LIBRARY_PATH`
5. 再执行实际业务命令

### KMVue Linux / kmconvertservice

1. 进入 `kmconvertservice` 所在目录
2. 直接启动转换服务
3. 查看是否缺少运行库或环境变量
4. 不要只按“加密卡连接不上”处理，先确认转换服务自己能启动
5. 转换服务正常后，再查加密卡、CLOUD 配置和网络

### Linux 加密卡

1. 执行 `KmPermitServer_Linux_Server.sh`
2. 或直接执行 `KmPermitServer_Linux`
3. 如果报缺少库文件，先补依赖或调整 `LD_LIBRARY_PATH`
4. 再检查授权文件、端口、防火墙和客户端连接

## 处理步骤

1. 先让对应程序在命令行下能直接启动
2. 解决运行库、Java、环境变量问题
3. 再检查业务配置、端口、防火墙和授权
4. 最后重启服务并验证业务功能

## 验证方式

- 程序能在命令行直接启动
- 日志中不再出现运行时缺失或依赖库缺失
- 对应服务进程存在
- 业务侧错误不再误报为连接失败
